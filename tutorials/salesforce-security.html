Salesforce Security Architecture — The Decisions That Keep CISOs Awake at Night

Salesforce's security model is sophisticated and layered — but its complexity means that misconfiguration is common and the consequences are severe. A deep analysis of the decisions that determine whether your Salesforce org is genuinely secure.

👤 Vishal Sharma ⏱ 23 min read 📅 2026 🎯 CISOs, Tech Leaders, Solution Architects, Delivery Managers

The uncomfortable reality: Salesforce provides the tools for a secure implementation. It does not guarantee one. The majority of Salesforce security incidents investigated by Salesforce's own security team originate from misconfiguration, not from platform vulnerabilities. Understanding the security model is not optional for tech leaders — it is a governance responsibility.

The Salesforce Security Model — Layered by Design

Salesforce implements security in layers, and understanding the layering is essential to understanding how breaches occur and how to prevent them.

From outermost to innermost:

  • Organisation security — IP restrictions, login hours, Multi-Factor Authentication enforcement, session settings, and trusted IP ranges
  • Object-level security — which objects (Account, Opportunity, Case) a user can access at all — controlled by Profiles and Permission Sets
  • Record-level security — which specific records within an object a user can access — controlled by the Sharing Model (OWD, Sharing Rules, Manual Sharing, Apex Managed Sharing)
  • Field-level security — which specific fields on a record a user can read or edit — controlled by Profiles and Permission Sets

A user with access to an object does not automatically have access to all records of that type. A user with access to a record does not automatically see all fields on that record. Every layer is a control point. Every misconfigured layer is a potential exposure.

Profiles, Permission Sets and the Principle of Least Privilege

Profiles are the foundation of Salesforce access control — every user must have exactly one Profile that defines their baseline object permissions, field-level security and system permissions. Permission Sets extend a Profile's access without replacing it, allowing granular additional permissions to be granted to specific users without creating Profile proliferation.

The principle of least privilege — granting users exactly the access they need, and no more — is the foundational principle of Salesforce security design. It is also the principle most frequently violated in practice, for understandable reasons: it is easier to grant broad access than to map detailed requirements, and sales cycles create pressure to make the system available quickly rather than securely.

Salesforce is moving architecturally away from Profiles towards a Permission Set-centric model. Permission Set Groups allow sets of Permission Sets to be assigned together, providing the convenience of Profile-style bundled permissions with the granularity and flexibility of Permission Sets. New implementations should design with Permission Sets and Permission Set Groups as primary access control mechanism, using the minimum required Profiles as baselines.

A governance principle: Every quarter, audit users with System Administrator profile. This should be the smallest possible set of people. System Administrator bypasses almost all security controls in Salesforce — its use should be logged, monitored and regularly reviewed.

The Sharing Model — The Most Complex Decision

The Salesforce sharing model determines which records each user can see and edit, independent of their object-level permissions. It operates through a hierarchy of mechanisms, from most restrictive to most permissive:

Organisation-Wide Defaults (OWD) set the baseline access for every object — Private (only the owner and hierarchy above can access), Public Read Only (everyone can view, only owner can edit), Public Read/Write (everyone can view and edit). OWD is the baseline from which all other sharing rules expand access — it cannot restrict access that has already been granted through another mechanism.

Role Hierarchy automatically grants managers access to their subordinates' records. This is a powerful and often misunderstood mechanism. A CEO at the top of the role hierarchy can access every record owned by every user below them in the hierarchy, for every object with OWD set to Private. Understanding who is at the top of your role hierarchy is a security question, not just an organisational one.

Sharing Rules extend access beyond the OWD baseline for defined groups of users. They can be criteria-based (share records matching specific field values) or ownership-based (share records owned by a specific group). Sharing Rules can only expand access, never restrict it.

The most common sharing model mistake is setting OWD too permissive as a shortcut, then attempting to restrict access through manual controls. Salesforce's sharing model is designed to expand from restrictive — starting permissive and adding restrictions is architecturally unsupported and creates unintended access.

Field-Level Security — Often Overlooked

Field-Level Security (FLS) controls whether a user can read or edit specific fields on records they have access to. It is configured through Profiles and Permission Sets and is independent of record-level sharing. A user with access to an Account record does not automatically see all fields on that Account — FLS determines which fields are visible and editable.

FLS is the most commonly overlooked layer of Salesforce security, with significant compliance implications:

  • PII fields — Social Security Numbers, date of birth, passport numbers — should have FLS configured to restrict visibility to only users with a documented business need
  • Financial data fields — annual revenue, credit limits, payment terms — often need FLS restriction in organisations with segregation of duties requirements
  • Sensitive service data — medical information in healthcare implementations, legal case details in professional services — have regulatory FLS requirements

Apex code that bypasses FLS is a common vulnerability. By default, Apex runs in system context and ignores FLS — it sees all fields regardless of the running user's permissions. Developers must explicitly call Schema.getGlobalDescribe() and check isAccessible() on fields before exposing them through custom code, or use the WITH SECURITY_ENFORCED SOQL clause.

API Security and Connected App Governance

Every Connected App registered in Salesforce represents an attack surface. Connected Apps use OAuth to grant external systems access to Salesforce data — a poorly governed Connected App estate means unknown systems have access to your CRM data through credentials that may not be actively monitored or rotated.

Critical API security questions every tech leader should be able to answer:

  • How many Connected Apps does your Salesforce org have? (Run a SOQL query: SELECT Id, Name, CreatedDate FROM ConnectedApplication)
  • When were each app's OAuth tokens last reviewed?
  • Are any Connected Apps using Username/Password OAuth flow? (This flow is deprecated and insecure — it should be replaced with JWT Bearer or Web Server flow)
  • Which Connected Apps have access to which OAuth scopes? An app with full scope has unrestricted access to all data the authorising user can access

Named Credentials should be used for all outbound API calls from Salesforce, storing authentication credentials in encrypted platform storage rather than in custom fields or custom metadata where they may be inadvertently exposed.

The Most Common Misconfigurations

These are the security misconfigurations that appear most frequently in Salesforce security audits:

  • MFA not enforced at org level — relying on users to opt in to Multi-Factor Authentication rather than enforcing it for all users
  • Excessive System Administrator profiles — more than 3-5 System Administrators in any org is a red flag; most organisations have far more than necessary
  • OWD set to Public Read/Write on sensitive objects — often set during initial implementation for convenience and never reviewed
  • Apex without FLS enforcement — custom code that queries and displays fields without checking FLS, exposing data to users who should not see it
  • Unreviewed Connected Apps — apps registered by former employees, forgotten integrations or development experiments that still have active OAuth access
  • Debug logs containing sensitive data — Salesforce debug logs can capture field values including sensitive data; uncontrolled debug log access is a data exposure risk
  • Guest User permissions too broad — Salesforce Experience Cloud's Guest User profile controls what unauthenticated visitors can access; overly permissive Guest User settings have caused significant data exposures
