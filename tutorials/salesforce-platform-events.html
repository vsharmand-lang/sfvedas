Salesforce Platform Events — Why Event-Driven Architecture Belongs at the Centre of Your Design

Platform Events are one of the most powerful and most underutilised capabilities in the Salesforce ecosystem. A deep technical and architectural analysis for teams building real-time, scalable Salesforce integrations.

👤 Vishal Sharma ⏱ 21 min read 📅 2026 🎯 Solution Architects, Tech Leads, Senior Developers

The architectural gap: Most Salesforce integrations are built synchronously — a user action triggers a callout, the callout completes, the user waits for a response. This pattern works for simple scenarios and breaks at scale. Platform Events offer an architectural alternative that is more resilient, more scalable and more maintainable — and is consistently chosen last rather than first.

The Publish-Subscribe Pattern

The publish-subscribe (pub-sub) pattern is one of the foundational patterns in distributed systems architecture. A publisher produces messages about events that have occurred — without knowing or caring who will consume those messages. Subscribers register interest in specific event types and receive messages when they are published — without knowing or caring who published them.

This decoupling is architecturally powerful because it enables publishers and subscribers to evolve independently. Adding a new system that needs to react to Salesforce events does not require any change to Salesforce — the new system simply subscribes to the relevant event. Removing a consuming system does not affect the publisher. The architecture is extensible by design.

Compare this to synchronous architecture: when Salesforce closes an Opportunity and makes synchronous callouts to three downstream systems, every downstream system's availability and performance directly affects the Salesforce user experience. If one callout times out, the Opportunity update fails or the user waits. Platform Events eliminate this coupling entirely.

Platform Events — The Mechanics

A Salesforce Platform Event is a custom object defined with the suffix __e. Like other Salesforce objects, it has fields that you define. Unlike other objects, its records are not stored persistently — they are published to the event bus and delivered to subscribers, then retained for 72 hours for replay purposes.

Publishing a Platform Event is done through Apex (EventBus.publish()), through Flow, or through the REST API from external systems. Subscribing is done through Apex triggers on the event object, through Flow, through the Streaming API (for external systems), or through the pub/sub API (Salesforce's newer, more scalable streaming protocol).

The key constraint: Platform Event publishing in Apex is asynchronous in the context of the database commit — events are published after the transaction commits successfully. This means a published event is guaranteed to have a committed transaction behind it, which is the correct behaviour for event-driven architecture.

Change Data Capture vs Platform Events

Salesforce offers two event-driven mechanisms that are frequently confused: Platform Events and Change Data Capture (CDC). Understanding the distinction is essential for correct architectural choices.

Change Data Capture automatically publishes events whenever a Salesforce record is created, updated, deleted or undeleted. These events are published by the platform without any custom code and contain the full changed field values. CDC is ideal for keeping external systems in synchronisation with Salesforce's system-of-record data — ERP synchronisation, data warehouse replication, external search index maintenance.

Platform Events are explicitly published by business logic — in Apex, Flow or via API. They represent business events ("Order Approved", "Customer Escalated", "Contract Signed") that carry business-specific payload, not just record field changes. Platform Events are ideal for triggering business process reactions across system boundaries.

The correct choice: use CDC when you need to keep data in sync; use Platform Events when you need to notify systems that a business event has occurred.

Designing for Scale with Platform Events

Platform Events are designed to scale — Salesforce processes millions of events per day across its customer base. The architectural considerations for high-volume Platform Event implementations:

  • Apex trigger governor limits — Platform Event triggers run in asynchronous context but still have governor limits. At high volume, consider whether processing should be in the trigger or delegated to a Queueable or Batch job
  • Event volume limits — Salesforce imposes daily limits on Platform Event publishes based on licence type. For very high-volume scenarios, validate that your event volume falls within your licence entitlement
  • External subscriber architecture — for external systems consuming Salesforce events at scale, the pub/sub API (gRPC-based) is more scalable than the legacy CometD Streaming API for high-throughput scenarios
  • ReplayId management — external subscribers should persist their ReplayId (event position in the bus) so that subscriber failures can recover from the last successfully processed event rather than re-requesting all events

Error Handling in Async Architectures

Error handling in event-driven architectures requires explicit design that does not arise naturally in synchronous architectures. When a callout fails, the failure is immediately visible and the transaction can be rolled back. When an event subscriber fails, the failure is invisible to the publisher, and the question of what happens to the unprocessed event must be answered by design.

The patterns for Platform Event error handling:

  • Dead-letter event management — events that cannot be processed after multiple retries should be written to a persistent store (a custom object, an external queue) for manual review and reprocessing
  • Idempotency — event consumers should be designed to handle the same event multiple times without producing duplicate effects. Network failures can cause event redelivery; consumer logic must handle this gracefully
  • Monitoring and alerting — Platform Event subscriber failures should trigger operational alerts. The Salesforce Event Monitoring log can track event delivery, but proactive alerting requires additional tooling

Real-World Patterns

These are the Platform Event patterns that deliver the most architectural value in production enterprise Salesforce implementations:

  • Order orchestration — Opportunity Close Won publishes a Platform Event; ERP creates an order, billing creates an invoice, fulfilment creates a shipment, and customer success creates an onboarding task — all asynchronously, all independently scalable
  • Real-time cross-cloud notification — Sales Cloud activity publishes events that Marketing Cloud consumes to suppress prospects from campaigns, update journey membership and trigger personalised communications
  • External system event reception — external systems publish events to Salesforce via REST API, triggering Salesforce business process without requiring the external system to call Salesforce synchronously during its own transaction
  • Audit and compliance streams — every significant business event published as a Platform Event creates an immutable audit trail that can be consumed by compliance and logging systems independently of the transactional system

  • ✦ Error handling in async architectures requires explicit dead-letter queue design.
  • ✦ Platform Events scale to millions of events per day at no additional licence cost.
