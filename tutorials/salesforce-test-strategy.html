Salesforce Testing Strategy — Why 75% Code Coverage Is the Floor, Not the Target

Salesforce requires 75% Apex test coverage before production deployment. Most teams treat this as the target. The organisations with stable, confident delivery treat it as the minimum. A deep look at what a mature Salesforce testing strategy actually looks like.

👤 Vishal Sharma ⏱ 20 min read 📅 2026 🎯 Tech Leads, Solution Architects, QA Engineers, Delivery Managers

The quality paradox: Salesforce's 75% coverage requirement is a platform safeguard against the worst implementations, not a quality standard for professional delivery. Meeting it confirms that someone wrote some tests. It says nothing about whether those tests are meaningful, whether they cover edge cases, or whether they would catch a regression.

The Code Coverage Myth

Code coverage measures which lines of code are executed during test runs. It does not measure whether those lines are tested correctly, whether the right outcomes are asserted, or whether the tests would detect meaningful regressions. An Apex test that calls a method, does not assert any outcomes, and simply verifies that no exception is thrown contributes to code coverage while providing almost no quality assurance value.

The organisations with genuinely high-quality Salesforce delivery target 85-90% code coverage with an additional quality criterion: meaningful assertion coverage. Every test method must assert specific outcomes — field values, record counts, exception types, callout parameters — not merely verify that no runtime error occurred.

The test quality litmus test: If you comment out the core logic in an Apex method and all tests still pass, your tests are not testing anything meaningful. Run this experiment on your most critical classes. The results are usually illuminating.

The Salesforce Test Pyramid

The test pyramid principle — many unit tests, fewer integration tests, even fewer end-to-end tests — applies to Salesforce with some platform-specific adaptations:

  • Unit tests — Apex unit tests covering individual classes and methods in isolation, using mock data and mock callouts. Fast, numerous, the foundation of quality confidence
  • Integration tests — tests that verify the interaction between multiple Salesforce components (triggers, handlers, services) with realistic data scenarios. Slower but essential for detecting component interaction bugs
  • System tests — full business process scenarios executed against a production-like sandbox, verifying end-to-end behaviour including UI, automation and integration. Slowest and most expensive to maintain

Most Salesforce teams have an inverted pyramid — few unit tests, extensive manual system testing. This produces slow, expensive, unreliable quality assurance that does not scale with delivery pace.

What Makes an Apex Test Actually Good

A high-quality Apex test has five characteristics:

  • Specific setup — creates only the data it needs for the scenario it tests, using test data factories that produce consistent, realistic data
  • Single behaviour — tests one specific behaviour or scenario per test method. Test methods that test multiple scenarios are harder to diagnose when they fail
  • Explicit assertions — asserts specific field values, record counts and method return values, not just absence of exceptions
  • Isolated execution — does not depend on data created by other tests or on org configuration that might vary between environments
  • Meaningful name — the test method name describes what scenario is being tested and what the expected outcome is. testOpportunityTrigger() is not a meaningful name. whenOpportunityCloses_shouldCreateOnboardingTask() is

UI and End-to-End Testing

Apex unit tests cover server-side logic but do not test the user interface, Lightning Web Components, page layouts or end-to-end business processes. These must be tested separately through UI and end-to-end testing approaches.

Salesforce has historically been challenging for automated UI testing because of its dynamic, JavaScript-rendered interface. The tooling landscape has improved significantly:

  • Selenium/WebDriver — the traditional approach, still viable but maintenance-intensive due to DOM changes with Salesforce releases
  • Salesforce CLI and Testing Frameworks — Jest for Lightning Web Component unit testing, enabling component-level UI testing without browser automation
  • Copado, Provar and similar platforms — commercial Salesforce-specific testing platforms that abstract the DOM complexity and provide more maintainable test scripts

Test Data Strategy

Test data management is one of the most significant practical challenges in Salesforce testing. The approaches:

Test Data Factories — Apex classes that create test records with appropriate default values, allowing test methods to request realistic test data without duplicating setup code across hundreds of test methods. Every mature Salesforce test suite should have a TestDataFactory or equivalent pattern.

@testSetup — the Salesforce annotation that executes a setup method once per test class and shares the created data across all test methods, reducing test execution time significantly for data-intensive test classes.

Static test data vs dynamic test data — tests that rely on specific records existing in the org (static data) are fragile and environment-dependent. Tests that create all required data dynamically in the test setup are portable and reliable. Always prefer dynamic data creation.

Testing in CI/CD Pipelines

The full value of a mature Salesforce test suite is realised only when tests run automatically on every code change, before deployment to any environment. This requires a CI/CD pipeline with Apex test execution as a mandatory gate.

Implementation approaches:

  • GitHub Actions / Azure DevOps / Bitbucket Pipelines — connect to Salesforce via JWT-based authentication, validate metadata against a scratch org or sandbox, run specified Apex tests, fail the pipeline if coverage or assertion thresholds are not met
  • Test run strategy — running all tests on every pull request is thorough but slow for large orgs. Selective test execution — running only tests related to changed classes — is faster but requires accurate dependency mapping
  • Test execution time management — long test execution times are the most common reason CI/CD pipelines are abandoned. Invest in parallel test execution and test class optimisation to keep pipeline execution under 15 minutes

Key Takeaways

  • ✦ 75% coverage is the legal minimum, not a quality target. Aim for 85%+ with meaningful assertions.
  • ✦ Tests with no assertions are technically coverage but practically worthless.
  • ✦ Test data factories are essential for maintainable test suites.
  • ✦ End-to-end testing on Salesforce requires tool investment — manual UAT does not scale.
  • ✦ Shift left — test earlier, test more often, test in CI/CD.
